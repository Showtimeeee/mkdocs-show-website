# Гайд по PDD: Puzzle Driven Development🧩

![pdd_logo](../images/pdd_logo.png)

**PDD (Puzzle Driven Development)** — это метод декомпозиции сложных задач

Вместо того чтобы пытаться решить большую задачу целиком (что долго и сложно), вы разбиваете её на множество маленьких «головоломок» прямо в коде, сдаёте основную функциональность быстро, а головоломки раздаёте команде.

Главная идея: **сложную задачу не нужно доделывать до конца перед слиянием в master. Достаточно сделать минимум, а всё остальное оформить как `@todo` и делегировать.**

---

## Ключевые концепции

- **Головоломка (Puzzle)**: Это `@todo`-комментарий в коде, который описывает, что именно нужно доработать, и ссылается на задачу в трекере (например, `#123`). Каждая головоломка — это маленькая, чётко сформулированная задача.

- **Минимальная реализация**: Вы пишете ровно столько кода, чтобы задача **уже работала** (компилировалась, проходила тесты, могла использоваться другими). Всё, что требует уточнения или улучшения, выносите в головоломки.

- **Декомпозиция через код**: Большая задача разбивается не в начале (когда много неизвестного), а в процессе написания кода. Дошли до сложного места — оставили головоломку, пошли дальше.

- **Асинхронное решение**: Головоломки могут решаться позже, любым разработчиком, независимо от автора исходного кода.

---

## Схема работы (Step-by-Step)

1. **Получили задачу.** Например: «Сделать класс `DBWriter` для записи данных в базу».

2. **Понимаем, что задача сложная.** Непонятно, какая база данных, нужны ли транзакции, как обрабатывать ошибки, нужна ли пакетная вставка. Решать всё сразу — значит застрять в ветке на неделю.

3. **Пишем минимальную версию.** Делаем просто: один метод `insert()`, без транзакций, без пакетной обработки. Код работает и его уже можно использовать.

4. **Оставляем головоломки.** В тех местах, где мы сделали упрощение, пишем `@todo`:

```java
/**
 * @todo #123 Сейчас данные вставляются по одной строке.
 *  Нужно добавить пакетную вставку (batch insert),
 *  иначе при большой нагрузке будет медленно.
 */
void insert(String text) {
    db.execute("INSERT INTO data VALUES ('" + text + "')");
}
```

5. **Сливаем в `master`.** Да, с недоделками. Ветка живёт не больше пары часов. Никто не ждёт.

6. **Головоломки становятся задачами.** Инструмент вроде [0pdd.com](https://0pdd.com) сам находит все `@todo` и создаёт из них отдельные задачи в трекере.

7. **Другие разработчики решают головоломки.** Кто-то берёт задачу «добавить пакетную вставку», исправляет код, убирает `@todo`, закрывает задачу.

---

## Сравнение: PDD vs обычный подход

| **Обычный подход** | **PDD** |
|---------------------|--------|
| Ветка живёт неделями, пока задача не будет сделана идеально | Ветка живёт часы, код мержится сразу |
| Один разработчик тащит всю сложность | Сложность разбита на мелкие головоломки, их можно раздать команде |
| Технический долг копится невидимо | Каждая недоработка — это явная `@todo` в коде и задача в трекере |
| Новые члены команды не знают, с чего начать | Головоломки — готовые маленькие задачи для входа в проект |

## Структура головоломки

Хорошая головоломка должна содержать:

| Что | Пример | Зачем |
|-----|--------|-------|
| Тег `@todo` | `@todo` | Чтобы легко найти все головоломки в коде |
| ID задачи | `#123` | Связь с трекером (Issue в GitHub, Jira) |
| Что нужно сделать | «Добавить пакетную вставку» | Чтобы понятно было, что требуется |
| Почему это важно | «иначе будет медленно» | Чтобы следующий понимал приоритет |

**Пример:**

```java
// @todo #123 Сейчас мы просто логируем ошибку. Нужно добавить retry с экспоненциальной задержкой, иначе при временных сбоях данные будут теряться.
catch (SQLException e) {
    log.error(e);
}
```

## Реальный пример (из open-source)

В проекте **EO** (язык программирования) была сложная задача [#2931](https://github.com/objectionary/eo/issues/2931) о поведении атрибута `ρ`.

Вместо того чтобы решать её целиком (фундаментальная проблема, много неизвестного), команда:

- Сделала базовую реализацию
- Оставила **38 головоломок** `@todo` в коде
- Слила изменения в `master`
- Решала эти головоломки **полгода**, по одной, разными разработчиками

**Результат:** Функциональность была в продакшене сразу, а сложные доработки шли параллельно, не блокируя никого.

## Автоматизация

Головоломки можно собирать вручную, но лучше использовать автоматические инструменты:

| Инструмент | Что делает |
|------------|------------|
| **[0pdd.com](https://0pdd.com)** | Сканирует GitHub-репозиторий, находит все `@todo #<ID>` и создаёт из них Issues |
| **PDD Ruby gem** | Проверяет головоломки локально перед коммитом |

## ❓ Вопросы с собеседований (ТОП-5)

### 1. Кто придумал PDD и в чём его суть?
**Ответ:** PDD предложил **Егор Бугаенко (Yegor Bugayenko)**. Суть в том, чтобы разбивать сложные задачи на маленькие головоломки (`@todo` в коде), быстро сдавать работающий код в `master`, а головоломки делегировать команде.

### 2. Чем PDD отличается от обычных `TODO`?
**Ответ:** Обычные `TODO` часто забываются. В PDD каждая головоломка привязана к задаче в трекере (`#123`), имеет чёткое описание и превращается в управляемую рабочую единицу.

### 3. Как PDD помогает с декомпозицией задач?
**Ответ:** Вместо того чтобы заранее придумывать, на какие части разбить большую задачу, вы разбиваете её прямо в коде — в процессе написания минимальной версии. Каждая `@todo` — это готовая маленькая задача.

### 4. Почему PDD позволяет быстрее сдавать код?
**Ответ:** Потому что не требует полной завершённости. Код должен быть **работоспособным**, но может содержать головоломки о будущих улучшениях. Это позволяет сливать изменения за часы, а не за недели.

### 5. Какие инструменты используются для PDD?
**Ответ:** Основной — **[0pdd.com](https://0pdd.com)**, который автоматически создаёт задачи из `@todo` в GitHub. Ещё есть **PDD Ruby gem** для локальной проверки.

---

## Заключение

PDD — это не про то, как писать код. Это про **как разбивать большую работу на маленькие куски и распределять их в команде**.

Вместо того чтобы неделю сидеть над сложной задачей, вы:
1. Делаете минимум
2. Оставляете головоломки
3. Сливаете код
4. Раздаёте головоломки команде

Все довольны: код уже в продакшене, а доделки идут параллельно.
