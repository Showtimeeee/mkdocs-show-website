    Гайд по Git

    Git — это распределенная система контроля версий, которая используется для отслеживания изменений в файлах и координации работы над проектами. Она широко используется в разработке программного обеспечения и является важным инструментом для специалистов по автоматизации качества (AQA). В этом руководстве мы рассмотрим основные команды и концепции Git, которые помогут вам эффективно использовать его в работе и на собеседованиях.

    Основные концепции Git

    Перед тем как начать использовать Git, важно понять его ключевые концепции:

    • Репозиторий (Repository): Хранилище вашего кода и истории изменений.
    • Коммит (Commit): Запись изменений в репозиторий.
    • Ветка (Branch): Отдельная линия разработки, позволяющая работать над новыми функциями или исправлениями независимо от основной ветки.
    • Слияние (Merge): Объединение изменений из одной ветки в другую.
    • Конфликт (Conflict): Ситуация, когда Git не может автоматически объединить изменения из разных веток.
    • Удаленный репозиторий (Remote Repository): Репозиторий, хранящийся на удаленном сервере (например, GitHub, GitLab).
    • HEAD: Указатель на текущий коммит (обычно на последний коммит в текущей ветке).

    Установка Git

    Для начала работы с Git вам нужно установить его на ваш компьютер. Скачайте и установите Git с официального сайта.

    Основные команды Git

    Настройка Git

    Настройка имени и email: 

    git config --global user.name "Ваше Имя"
git config --global user.email "ваш.email@example.com"

    Работа с репозиторием и коммитами

    Команда Описание
    git init Создать новый локальный репозиторий
    git clone <URL> Клонировать удаленный репозиторий
    git status Проверить статус репозитория
    git add <файл> Добавить файл в индекс
    git add . Добавить все изменения в индекс
    git commit -m "сообщение" Создать коммит с сообщением
    git log Просмотреть историю коммитов
    git show <hash> Просмотреть изменения в конкретном коммите
    git diff Показать изменения в файлах, которые еще не добавлены в индекс

    Работа с ветками

    Команда Описание
    git branch Показать список локальных веток
    git branch <имя> Создать новую ветку
    git checkout <имя> Переключиться на другую ветку
    git checkout -b <имя> Создать и переключиться на новую ветку
    git merge <имя> Слить изменения из указанной ветки в текущую
    git branch -d <имя> Удалить ветку (если она уже слита)
    git branch -D <имя> Принудительно удалить ветку (даже если не слита)

    Работа с удаленными репозиториями

    Команда Описание
    git remote add origin <URL> Добавить удаленный репозиторий с именем origin
    git push origin <имя> Отправить изменения в удаленный репозиторий
    git pull origin <имя> Получить изменения из удаленного репозитория и смержить их
    git fetch origin Получить изменения из удаленного репозитория без слияния
    git remote -v Показать список удаленных репозиториев

    Отмена изменений и настройка

    Команда Описание
    git revert <hash> Создать новый коммит, который отменяет изменения указанного коммита
    git reset --hard <hash> Сбросить состояние репозитория до указанного коммита (удалить все после)
    git reset <файл> Убрать файл из индекса (после git add, но до коммита)
    git config --global user.name "Имя" Настроить имя пользователя
    git config --global user.email "email@example.com" Настроить email пользователя
    git config --list Проверить текущие настройки

    Работа с историей и продвинутые команды

    Команда Описание
    git rebase <ветка> Перебазировать текущую ветку на указанную. Переносит коммиты текущей ветки в конец целевой ветки.
    git stash Временно сохранить незакоммиченные изменения и очистить рабочую директорию
    git stash pop Восстановить последние изменения из stash и удалить их из stash
    git cherry-pick <hash> Применить изменения из указанного коммита в текущую ветку

    🚀 Разбор команды: git rebase

    git rebase (перебазирование) — это команда, которая позволяет перенести изменения из одной ветки в другую, но делает это более "чисто", чем merge.

    Как это работает: Допустим, вы создали ветку feature от main, и пока вы работали, в main кто-то добавил новые коммиты. * git merge создаст дополнительный "коммит слияния", который объединит истории. * git rebase возьмет ваши коммиты из feature и "переиграет" их поверх новых коммитов в main. История станет линейной и чистой, как будто вы только что создали ветку.

    Когда использовать: * Для поддержания чистой и линейной истории в репозитории. * Перед отправкой Pull Request (чтобы ваша ветка не отставала от основной).

    Важное предупреждение: Никогда не делайте rebase веток, которые кто-то кроме вас использует (например, общую ветку main или ветку коллеги). Это меняет историю и создаст хаос у других разработчиков.

    ❓ Вопросы с собеседований (ТОП-10)

    Этот раздел поможет вам подготовиться к техническому интервью. Вопросы охватывают как базовые концепции, так и реальные рабочие ситуации.

    1. В чем разница между git merge и git rebase?

    Ответ: * git merge объединяет две ветки, создавая новый "коммит слияния". Он сохраняет полную историю и показывает, когда и какие ветки сливались. Это безопаснее для истории, которая уже есть на сервере. * git rebase переписывает историю текущей ветки, перенося ее коммиты в конец целевой ветки. Это создает линейную, чистую историю без лишних коммитов слияния, но не стоит применять к веткам, с которыми работают другие люди.

    2. Как разрешить конфликт в Git?

    Ответ: 1. Git пометит конфликтующие файлы. Вы увидите это в статусе (git status) — both modified. 2. Открыть конфликтующий файл. Git отметит участки конфликта так: 

    <<<<<<< HEAD
ваш код из текущей ветки
=======
код из вливаемой ветки
>>>>>>> feature-branch
    3. Вручную отредактировать файл, оставив нужный код и удалив маркеры (<<<, ===, >>>). 4. Сохранить файл, добавить его в индекс (git add <файл>). 5. Продолжить операцию: если был merge — сделать git commit, если rebase — git rebase --continue.

    3. Чем отличается git pull от git fetch?

    Ответ: * git fetch загружает новые данные из удаленного репозитория, но не сливает их с вашей локальной веткой. Ваша рабочая копия остается нетронутой. Это безопасный способ посмотреть, что нового у коллег. * git pull делает сразу две вещи: сначала git fetch (загружает данные), а затем git merge (пытается влить эти изменения в вашу текущую ветку).

    4. Что такое .gitignore и зачем он нужен?

    Ответ: Это специальный файл, в котором перечисляются шаблоны файлов и папок, которые Git должен игнорировать. Это нужно, чтобы не засорять репозиторий служебными файлами (папки IDE .idea/), временными файлами, кешем, виртуальными окружениями (venv/, node_modules/), файлами с паролями и т.д.

    5. Ты случайно сделал commit не в ту ветку. Как это исправить?

    Ответ: Ситуация: нужно было коммитить в feature, а закоммитили в main. 1. Переключиться на правильную ветку: git checkout feature 2. Забрать последний коммит из main: git cherry-pick main (или используя хеш коммита). 3. Вернуться на ветку main и откатить ее на коммит назад, удалив "ненужный" коммит: git checkout main и git reset --hard HEAD~1.

    6. Что такое git stash и когда его применяют?

    Ответ: git stash позволяет временно "отложить" незакоммиченные изменения в сторону и вернуть рабочую директорию к чистому состоянию (как в последнем коммите). Это полезно, когда вам нужно срочно переключиться на другую ветку или задачу, но вы не хотите коммитить недоделанную работу. Потом можно вернуть изменения командой git stash pop.

    7. Как исправить сообщение последнего коммита, если вы ошиблись?

    Ответ: Нужно использовать команду git commit --amend -m "Новое правильное сообщение". Эта команда заменит последний коммит новым (с новым хешем). Если коммит уже был запушен на сервер, после amend потребуется делать git push --force-with-lease (с осторожностью!).

    8. Объясни, что такое HEAD в Git.

    Ответ: HEAD — это указатель (ссылка) на текущий коммит или ветку, в которой вы сейчас находитесь. Обычно он указывает на имя ветки (например, refs/heads/main), а ветка уже указывает на конкретный коммит. Если переключиться на конкретный коммит (а не ветку), Git сообщит, что вы в состоянии "detached HEAD".

    9. Как удалить файл из репозитория, но оставить его на диске?

    Ответ: Обычно git rm <файл> удаляет файл и из Git, и с диска. Чтобы убрать файл только из-под версионного контроля, но оставить его в папке проекта (например, для локальных настроек), нужно использовать флаг --cached: git rm --cached <файл> После этого нужно добавить этот файл в .gitignore, чтобы Git снова не начал его отслеживать.

    10. Какие стратегии ветвления вы знаете (Git Flow, GitHub Flow)?

    Ответ: * Git Flow: Сложная стратегия с долгоживущими ветками (develop, main) и множеством вспомогательных (feature/*, release/*, hotfix/*). Подходит для проектов с четким циклом релизов. * GitHub Flow: Простая стратегия. Есть одна главная ветка (main) и ветки фич. После завершения работы над фичей создается Pull Request. После ревью и проверок ветка сливается в main и сразу деплоится. Подходит для веб-проектов и непрерывной доставки (CI/CD). * GitLab Flow: Вариация на тему, которая добавляет среды (окружения) или ветки для синхронизации с issues.
